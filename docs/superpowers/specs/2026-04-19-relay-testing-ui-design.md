# Relay Testing Workbench UI 重构设计稿

> 设计日期：2026-04-19  
> 适用项目：`H:\nettest`  
> 适用范围：`H:\nettest\NetTest.App\MainWindow.xaml` 及相关 ViewModel 交互重组  
> 当前阶段：设计确认稿（待用户审阅后进入实现计划）

## 1. 背景与目标

当前应用已经具备较完整的中转站测试、网络检测与报告导出能力，但界面组织方式更偏“技术模块堆叠”，而不是“用户任务驱动”。

当前问题主要体现在：

- 一级入口过多，且以技术模块命名，不以任务命名。
- 用户很难快速判断应该先点哪个功能。
- 单站测试、批量对比、深度测试之间的关系不够直观。
- 网络相关工具虽然有价值，但在视觉与导航层级上抢占了主功能位置。
- 页面整体偏高、偏松，卡片与留白过大，导致信息密度不足。

本次 UI 重构的目标不是新增能力，而是重组现有能力，让产品表达更准确：

> 这是一个“中转站测试工作台”，核心任务是测试单个中转站、批量对比多个中转站，并在结果异常时进行网络复核。

## 2. 核心产品定位

### 2.1 产品主线

本应用的主线能力只有两条：

1. **单站测试**：测试一个中转站当前是否可用、快不快、稳不稳、能不能正式接入。
2. **批量对比**：对多个中转站进行快速筛选、排序、选优，并支持对用户勾选的站点继续做深度测试。

### 2.2 辅助能力

网络相关能力不是主功能，而是辅助能力：

- 当单站测试或批量对比出现异常时，用于排查是本地网络问题，还是中转站本身问题。
- 因此网络页应该被定位为“**网络复核**”，不是首页，也不是默认主入口。

### 2.3 深度测试定位

“深度测试”不再是独立一级页面，而是并入主测试流程：

- 在 **单站测试** 中，它是单站的高级测试模式。
- 在 **批量对比** 中，它是快速筛选之后的第二阶段动作。

## 3. 信息架构

### 3.1 一级导航

左侧采用纵向侧边栏，一级导航收敛为 4 项：

1. **单站测试**（默认打开）
2. **批量对比**
3. **网络复核**
4. **历史报告**

### 3.2 明确移除的结构

以下结构从新设计中移除：

- 首页
- 总览页
- 独立的“中转站深度检测”一级页
- 把所有网络模块平铺成一级标签的方式

### 3.3 旧结构到新结构映射

| 当前结构 | 新结构 |
|---|---|
| 总览 | 移除 |
| 中转站检测 | 并入“单站测试” |
| 中转站深度检测 | 并入“单站测试”与“批量对比” |
| 网络检测 | 并入“网络复核” |
| 官方 API 可用性 | 并入“网络复核” |
| NAT / STUN | 并入“网络复核”的高级复核 |
| 测速 | 并入“网络复核”的性能复核 |
| 路由与 MTR | 并入“网络复核”的性能复核 |
| IP 与分流检测 | 并入“网络复核”的出口复核 |
| 端口扫描 | 并入“网络复核”的高级复核 |
| 历史记录 | 调整为“历史报告” |

## 4. 全局 UI 规则

### 4.1 布局规则

- 使用左侧纵向导航，不使用顶部主标签栏。
- 应用启动后直接进入 **单站测试**，不经过首页或总览页。
- 所有主页面都遵循一致的三层结构：
  - 顶部：页面标题与当前状态摘要
  - 中部：当前页面的主工作区
  - 右侧或下方：结果摘要与统一输出

### 4.2 视觉密度规则

本次重构采用更紧凑的桌面工作台风格：

- 字体整体缩小一档。
- 卡片内边距缩小。
- 区块间垂直留白压缩。
- 同层卡片间距减小。
- 结果区尽量短而密，不做大面积留白。
- 深度测试与高级项默认折叠，避免默认态过高。

### 4.3 视觉风格规则

- 深色专业工作台风格。
- 高信息密度，但层级必须明确。
- 图表用于展示，不与主要操作行为混用。
- 推荐与建议动作要先于原始结果展示。
- 状态色必须辅助文字，不允许只靠颜色表达结论。

## 5. 单站测试页设计

### 5.1 页面职责

单站测试页是应用默认打开页。

它只回答一个问题：

> 当前这个中转站，能不能用、稳不稳、值不值得继续用或继续深测。

### 5.2 页面三段式结构

单站测试页固定分为三段：

#### 1）当前站点配置

职责：只负责“测哪个站”。

内容：

- 目标地址
- 模型
- API Key
- 拉取模型按钮

约束：

- 不放模式切换
- 不放深度测试项
- 不放结果输出

#### 2）当前模式参数与执行

职责：只负责“怎么测”。

模式切换：

- 快速测试
- 稳定性测试
- 深度测试

默认模式：**快速测试**

行为规则：

- 快速测试默认展开。
- 稳定性测试仅在切换后显示轮次、间隔等参数。
- 深度测试仅在切换后显示高级测试项。
- 此区不重复地址、模型、Key 等基础配置。

#### 3）结果与统一输出

职责：只负责“测完看什么”。

内容：

- 当前模式下的结果摘要
- 关键指标（如 TTFT、流式速率、成功率）
- 统一输出日志
- 当前建议动作

### 5.3 单站测试模式定义

#### 快速测试

用途：默认入口。

目标：

- 快速判断当前站点能否正常工作。
- 给出基础性能与可用性结论。

默认显示：

- 是否可用
- TTFT
- 流式速率
- 基础结论

#### 稳定性测试

用途：观察一段时间内是否稳定。

目标：

- 看成功率
- 看波动
- 看连续失败

仅在切换后展开：

- 轮次
- 间隔
- 失败统计

#### 深度测试

用途：验证这个站点能否正式接入。

目标：

- 协议兼容
- 流式完整性
- 官方对照
- 多模态
- 缓存隔离
- 其他高级兼容性问题

规则：

- 不单独开一级页。
- 只在深度模式下展开高级项。
- 默认快速测试模式下保持折叠。

## 6. 批量对比页设计

### 6.1 页面职责

批量对比页回答的问题是：

> 多个中转站里，哪个最好、最快、最稳；哪些站值得继续做深度测试。

### 6.2 页面默认视图

批量对比页默认显示 **排行榜视图**。

不默认显示：

- 配置大表单
- 深度测试候选区
- 自动勾选结果

### 6.3 正式流程

批量对比页的正式流程固定为：

1. **导入或选择入口组**
2. **运行一次快速对比**
3. **生成排行榜图表与排行榜列表**
4. **由用户在排行榜列表项里手动勾选站点**
5. **对已勾选站点发起深度测试**

### 6.4 图表区规则

排行榜图表区只负责：

- 展示 Top 排名
- 展示综合分差距
- 帮助用户理解排序结果

明确禁止：

- 在图表柱条上点选
- 在图表区做勾选交互
- 让图表兼具展示与操作双重语义

### 6.5 列表区规则

排行榜列表项负责：

- 呈现每个站点的快速对比结果
- 承载用户勾选行为

明确规则：

- 只能勾选排行榜列表项
- 快速对比完成后，**默认一个都不勾选**
- 用户必须手动选择哪些站点进入深度测试

原因：

- 排名只是建议，不应自动替代用户决策。
- 推荐站点不等于自动进入深度测试。

### 6.6 批量深度测试定位

批量中的深度测试不再是单独一级页面，而是批量对比的第二阶段。

它的作用是：

- 对用户勾选的站点做更重的兼容性与质量验证。
- 帮助用户在 Top 结果里进一步筛出真正适合长期使用或正式接入的站点。

## 7. 网络复核页设计

### 7.1 页面职责

网络复核页不是主测试页，而是辅助排查页。

它只在以下情况下进入：

- 单站测试异常
- 批量对比结果异常
- 怀疑本地网络影响了中转站测试结果

### 7.2 页面核心问题

网络复核页要先回答：

> 当前异常更像本地网络问题，还是中转站本身问题？

### 7.3 页面组织方式

不再平铺所有网络模块，而是先让用户选择异常类型，再缩小排查范围。

推荐流程：

1. 先选择异常类型
   - 中转站不可用
   - TTFT 很高
   - 波动大
   - 地区 / 解锁异常
   - DNS / 分流怀疑
2. 再推荐对应复核模块

### 7.4 模块分层

#### 基础复核

- 网络检测
- 官方 API 可用性

适用于：

- 先排除本机网络明显异常
- 先确认官方链路是否正常

#### 性能复核

- 测速
- 路由 / MTR

适用于：

- 延迟高
- 波动大
- 怀疑链路质量异常

#### 出口复核

- IP 与分流检测

适用于：

- 出口异常
- DNS 解析异常
- 分流逻辑怀疑有误

#### 高级复核

- NAT / STUN
- 端口扫描

适用于：

- 更高级的网络定位问题
- 不作为默认第一步

### 7.5 输出规则

网络复核页必须先给结论，再给明细。

优先展示：

- 当前判断
- 推荐下一步
- 是否更像本地网络问题

而不是一上来就展示大量原始输出。

## 8. 历史报告页设计

### 8.1 页面职责

历史报告页用于沉淀与回看：

- 单站测试历史
- 批量对比历史
- 导出报告
- 最近归档

### 8.2 页面原则

- 不承担当前测试入口职责。
- 优先支持按时间和任务类型回看结果。
- 保留导出能力，但不抢占主测试流程的视觉位置。

## 9. 术语统一

为避免命名混乱，页面内统一采用以下术语：

| 术语 | 含义 |
|---|---|
| 单站测试 | 测一个中转站 |
| 批量对比 | 测多个中转站并排序 |
| 快速测试 / 快速对比 | 轻量、第一阶段测试 |
| 稳定性测试 | 多轮重复测试 |
| 深度测试 | 更重的兼容性与质量验证 |
| 网络复核 | 当测试异常时做辅助排查 |
| 排行图表 | 只读展示区 |
| 排行列表 | 可勾选的交互区 |

## 10. 明确不做的事情

本次设计阶段明确不做：

- 不新增新的网络检测能力。
- 不新增新的中转站测试能力。
- 不把图表做成交互主入口。
- 不保留首页或总览页。
- 不把深度测试继续保持为独立一级页面。

## 11. 实施优先级建议

建议实现顺序如下：

1. **一级导航重组**
   - 去掉首页 / 总览
   - 收敛为 4 个一级入口
2. **单站测试页重组**
   - 落地三段式结构
   - 默认快速测试
   - 深度项折叠
3. **批量对比页重组**
   - 默认排行榜视图
   - 图表只读
   - 列表手动勾选
4. **网络复核页重组**
   - 从工具集合改为辅助排查页
5. **历史报告页整理**
   - 统一历史与导出入口
6. **全局密度收紧**
   - 字体、卡片、间距统一缩紧

## 12. 设计确认结论

本次设计确认后的最终结论如下：

- 应用不再设置首页。
- 应用启动后直接进入 **单站测试**。
- 左侧使用纵向侧边栏导航。
- 一级导航固定为：**单站测试 / 批量对比 / 网络复核 / 历史报告**。
- **深度测试**并入单站测试与批量对比，不再独立成页。
- 批量对比固定流程为：**入口组 → 快速对比 → 排行榜列表勾选 → 深度测试**。
- 批量对比中，**图表只读，列表可勾选，默认不自动勾选任何项**。
- 网络功能整体降级为辅助复核能力。
- 全局视觉改为更紧凑、更高密度的专业工作台风格。
