# RelayBench 大厂级 UI 施工方向规划

> 版本：v1.0  
> 日期：2026-04-30  
> 状态：待审  
> 使用技能：`ui-ux-pro-max`、`brainstorming`、`chinese-documentation`  
> 关系：本文件是 UI 方向稿，配合 `docs/plans/2026-04-30-dashboard-shell-redesign.md` 作为后续施工依据。

## 1. 设计判断

`ui-ux-pro-max` 检索给出的关键结论是：

- 产品类型应按 **B2B SaaS / Analytics Dashboard / Real-Time Monitoring** 处理。
- 风格应优先采用 **Data-Dense Dashboard**、**Real-Time Monitoring**、**Swiss Modernism 2.0**。
- 色彩应使用 **Professional blue + neutral grey**，少量状态色用于成功、警告、错误和实时更新。
- 图表应重点使用 **Line Chart**、**Streaming Area Chart**、**Bar Chart**、**Radar / Parallel Coordinates**。
- UX 重点是 loading 状态、空状态、层级管理、焦点可见、减少无意义连续动画。

因此，RelayBench 不适合做成偏营销、偏装饰、偏消费级的“大圆角渐变卡片”。更合适的方向是：

> 克制、密集、清晰、可信、状态完整，像一个成熟的企业诊断工作台。

## 2. 大厂感来自什么

大厂产品的 UI 不只是“更好看”，核心是更稳定、更统一、更少让用户猜。

RelayBench 需要重点补齐 6 件事：

- **统一的信息架构：** 页面、子菜单、弹窗、抽屉、Toast 都有固定位置和视觉层级。
- **统一的设计 Token：** 颜色、字号、间距、圆角、阴影、动效不再各写各的。
- **清晰的任务路径：** 每页只能有 1 个主操作，其它都降级为次级工具。
- **专业的状态反馈：** 加载中、成功、失败、警告、空状态、禁用态、运行中都有专门样式。
- **高信息密度但不拥挤：** 表格、指标、日志、图表需要紧凑，但分组边界清楚。
- **可访问与可维护：** 焦点态、键盘操作、对比度、低动效模式、z-index 层级都要成体系。

## 3. 三种方向方案

### 方案 A：企业后台 Pro 风格

关键词：Ant Design Pro、阿里云控制台、腾讯云控制台、企业管理后台。

特点：

- 顶部导航 + 左侧菜单 + 内容卡片。
- 表格、筛选、标签、弹窗非常规范。
- 视觉克制，状态清晰。

优点：

- 最稳，最像企业级产品。
- WPF 落地成本低。
- 适合应用接入、批量评测、历史报告。

缺点：

- 如果处理不好，会显得普通后台化。
- 大模型对话和实时监控的科技感不够。

### 方案 B：运维监控台风格

关键词：Datadog、Grafana Cloud、Cloudflare Dashboard、Sentry。

特点：

- 强调实时状态、图表、日志、告警、事件流。
- 信息密度高。
- 深色局部日志面板和实时图表较多。

优点：

- 非常适合单站测试、网络复核、批量深测。
- 能强化“专业诊断工具”的定位。

缺点：

- 如果全局过暗，会压迫感强。
- 表单和应用接入场景不如企业后台舒服。

### 方案 C：AI 诊断工作台混合风格（推荐）

关键词：企业后台 Pro + 运维监控台 + OpenWebUI 式对话工作台。

特点：

- 外壳采用企业后台 Pro 的稳定结构。
- 单站、批量、网络采用运维监控台的信息呈现。
- 大模型对话采用 OpenWebUI 式中间主聊天区 + 右侧参数抽屉。
- 应用接入采用软件资产卡片。
- 历史报告采用时间线 + 详情。

推荐原因：

- 最贴合 RelayBench 的真实功能。
- 既有大厂后台的秩序，又有诊断工具的专业感。
- 不需要推翻现有 WPF 架构，可以逐页施工。

## 4. 最终视觉定位

建议定位为：

> Enterprise AI & Network Diagnostics Workbench  
> 企业级 AI 与网络接口诊断工作台

中文感受应该是：

- 不是小工具。
- 不是玩具化 Demo。
- 不是网页套壳。
- 像一个可长期使用、可发布给真实用户的专业软件。

## 5. 设计系统方向

### 5.1 色彩

建议使用“专业蓝 + 中性灰 + 状态色”：

| 用途 | 颜色 | 说明 |
|------|------|------|
| 主色 | `#2563EB` | 操作、选中、链接、主按钮 |
| 深主色 | `#1E40AF` | 重点标题、图表主线、深色 hover |
| 背景 | `#F8FAFC` | 全局背景，替代过重渐变 |
| 页面底色 | `#F1F5F9` | 工作台主体浅灰蓝 |
| 卡片 | `#FFFFFF` | 一级内容卡片 |
| 强卡片 | `#FBFDFF` | 顶部栏、弹窗、抽屉 |
| 边框 | `#E2E8F0` | 默认分隔线 |
| 主文本 | `#0F172A` | 高对比正文 |
| 次文本 | `#475569` | 次级说明，避免过浅 |
| 成功 | `#16A34A` | 通过、完成、可用 |
| 警告 | `#F59E0B` | 兼容性未知、需要注意 |
| 错误 | `#DC2626` | 失败、危险操作 |
| 实时 | `#0EA5E9` | 流式、运行中、刷新中 |

施工要求：

- 不再把紫蓝渐变作为主视觉大面积使用。
- 渐变只保留在 Logo、主按钮 hover、进度条这类小面积元素。
- 图表颜色用蓝、绿、橙、红固定语义，不随页面随意变化。

### 5.2 字体

WPF 桌面应用建议：

- 中文：`Microsoft YaHei UI` 或 `Microsoft YaHei`。
- 英文和数字：可使用 `Segoe UI Variable`，如果兼容性不稳则继续使用 `Microsoft YaHei UI`。
- 代码和日志：`Cascadia Code`，没有则回退 `Consolas`。

字号建议：

| 位置 | 字号 | 字重 |
|------|------|------|
| 页面标题 | 18px | SemiBold |
| 卡片标题 | 14-15px | SemiBold |
| 正文 | 12.5-13px | Regular |
| 表格正文 | 12px | Regular |
| 标签/说明 | 11.5-12px | Regular |
| 指标大数字 | 22-28px | SemiBold |
| 日志/代码 | 12px | Regular |

施工要求：

- 不按窗口宽度动态缩放字体。
- 不使用负字距。
- 所有按钮文字必须完整显示，不能靠缩小到很小来硬塞。

### 5.3 间距

采用 4px / 8px 基线：

| 用途 | 建议 |
|------|------|
| 页面外边距 | 12px |
| 卡片间距 | 8-12px |
| 卡片内边距 | 12-16px |
| 表格行高 | 36-40px |
| 工具条高度 | 40-44px |
| 顶部栏高度 | 48-52px |
| 左侧导航宽度 | 220-240px |
| 抽屉宽度 | 380-420px |

施工要求：

- 表格、日志、图表区域用紧凑间距。
- 页面级区域保留足够呼吸感。
- 卡片不再层层套卡片。

### 5.4 圆角和阴影

大厂后台一般不靠厚重阴影体现品质，而靠层级和边界。

建议：

- 小控件圆角：6-8px。
- 输入框和按钮：8px。
- 卡片：10-12px。
- 弹窗和抽屉：14-16px。
- 不再到处使用 20px 以上大圆角。
- 阴影只给浮层、导航栏、弹窗；普通卡片以边框为主。

## 6. 页面布局新方向

### 6.1 全局 Shell

目标：像成熟控制台，而不是普通窗口里塞页面。

结构：

```text
TitleBar
├─ Brand
├─ Global Task Progress
└─ Window Controls

Workbench
├─ Navigation Rail
└─ Page Workspace
   ├─ Context Header
   ├─ Command Bar
   ├─ Main Content
   └─ Status Bar
```

改法：

- 页面标题和页面说明放进统一 `Context Header`。
- 当前接口、当前模型、运行状态这些跨页面上下文放到 `Command Bar` 或状态胶囊。
- 主操作永远放右侧。
- 左侧导航只做导航，不承载大段说明。

### 6.2 页面命令区

每个页面都使用统一命令区：

```text
Command Bar
├─ 当前上下文摘要
├─ 二级工具：历史接口 / 图表 / 参数 / 导入 / 导出
└─ 主操作：开始测试 / 发送 / 开始扫描 / 应用 / 导出
```

好处：

- 用户视线稳定。
- 不会每页都重新学习按钮位置。
- 主操作更像大厂产品的“工作流入口”。

### 6.3 右侧抽屉体系

适合放入抽屉的内容：

- 大模型对话参数。
- 单站测试能力模型。
- 网络复核高级参数。
- 应用接入写入预览。
- 历史报告筛选条件。

抽屉规则：

- 右侧滑出，宽 380-420px。
- 顶部固定标题和关闭按钮。
- 中间滚动。
- 底部可放保存/应用/重置按钮。
- 点击左侧空白关闭，但点击列表项和输入控件不关闭。

## 7. 各页面大厂化方向

### 7.1 单站测试

目标：从“测试表单”升级为“接口健康工作台”。

新结构：

- 顶部命令区：当前接口、历史接口、拉取模型、查看图表、开始测试、停止。
- KPI 行：成功率、平均延迟、TTFT、吞吐、协议支持。
- 主体左侧：实时图表和并发分档。
- 主体右侧：检测明细和日志。
- 底部：能力矩阵。
- 右侧抽屉：能力模型和高级测试项。

更好看的改法：

- 测试模式不要用普通下拉，改成横向模式卡：快速、深度、连续。
- 测试项不要用密集 CheckBox，改成开关卡片，每张卡有名称、说明、状态。
- 图表卡 hover 显示放大按钮。
- 检测明细用状态徽标，不使用大段纯文本。

### 7.2 大模型对话

目标：从“内置聊天页”升级为“AI 对话工作台”。

新结构：

- 左侧：会话列表。
- 中间：聊天消息和输入区。
- 右侧：默认收起的参数抽屉。
- 顶部命令区：当前模型、多模型摘要、参数、预设、新建会话。

更好看的改法：

- 用户消息右侧、助手消息左侧，气泡不要超过 72% 宽度。
- 多模型回答采用列视图，每列类似评测面板。
- 输入区 sticky 在底部，附件条在输入框上方。
- 参数抽屉分组：接口、预设、生成参数、思考强度、多模型、会话操作。
- 思考强度改成分段按钮，而不是普通下拉。

### 7.3 批量评测

目标：从“批量表格”升级为“模型/接口排行榜”。

新结构：

- 顶部流程：入口组 → 快测 → 候选 → 深测。
- KPI 行：入口数、完成数、平均成功率、最佳候选、异常数。
- 快测排行榜：表格 + 条形指标。
- 候选池：可勾选卡片。
- 深测区：对比图、徽标、结论。

更好看的改法：

- 排行榜前 3 名使用更明显的排名徽标，但不要用夸张金银铜。
- 指标列增加小条形进度，提升可扫读性。
- 应用按钮改成行内图标按钮，hover 时出现文案。
- 入口组维护弹窗做成左右主从布局，底部固定操作栏。

### 7.4 应用接入

目标：从“检测列表”升级为“本机 AI 软件资产面板”。

新结构：

- 顶部：扫描状态和开始扫描。
- KPI 行：发现应用、可写入、可还原、异常项。
- 应用卡片网格：Codex、Claude CLI、OpenAI 兼容客户端等。
- 当前接口卡：历史接口、拉取模型、协议探测、应用当前接口。
- 扫描结果：概览、细节、支持范围。

更好看的改法：

- 应用卡片左上角使用统一线性图标，不使用 emoji。
- 协议用彩色 badge：Responses、OpenAI Chat、Anthropic。
- “原始 Trace”放成小图标按钮。
- “写入预览”改成代码预览块，可复制。
- 应用到软件弹窗顶部展示协议探测雷达或三段状态条。

### 7.5 网络复核

目标：从“多工具合集”升级为“网络诊断中心”。

新结构：

- 工具分组：基础复核、性能复核、出口复核、连接复核。
- 当前工具配置区。
- 当前工具结果区。
- 原始日志/Trace 区。

更好看的改法：

- 工具选择用工具卡矩阵，不用普通列表。
- 路由 / MTR 用时间线和跳点卡。
- IP 风险用风险评分卡 + 多源矩阵。
- 端口扫描用端口资产表，开放端口高亮。
- 原始日志统一深色控制台。

### 7.6 历史报告

目标：从“历史文本列表”升级为“诊断记录中心”。

新结构：

- 左侧时间线。
- 右侧报告详情。
- 顶部筛选：类型、日期、关键词。
- 右上角导出。

更好看的改法：

- 历史项使用时间线样式，显示类型标签和状态。
- 详情顶部显示摘要卡。
- 报告正文使用 Markdown 风格排版。
- 导出成功后使用 Toast，附带打开目录/复制路径。

## 8. 图表系统方向

图表不是装饰，是诊断证据。

建议：

| 场景 | 图表 |
|------|------|
| 实时延迟 / TTFT | Line Chart / Streaming Area Chart |
| 吞吐趋势 | Area Chart |
| 候选排行 | Horizontal Bar Chart |
| 多能力对比 | Radar Chart 或 Parallel Coordinates |
| 分档并发 | Step Line + Badge |
| IP 风险矩阵 | Data Table + Heat Color |
| 路由跳点 | Timeline / Path List |

施工要求：

- 每个图表必须有标题、图例、单位。
- 不能只靠颜色区分状态，必须有标签或图案。
- 实时图表必须有暂停或停止入口。
- 图表加载时显示 skeleton 或扫光。
- 图表空状态要说明“运行测试后显示”。

## 9. 交互和动效方向

大厂感不是动画多，而是动效服务状态。

保留：

- hover 颜色变化。
- 按钮按下反馈。
- 抽屉滑入。
- 弹窗淡入。
- 图表 loading 扫光。
- 实时状态点轻微 pulse。

减少：

- 大面积 scale 动画。
- 装饰性无限动画。
- 太重的阴影 hover。
- 不必要的渐变切换。

统一时长：

- hover：150ms。
- 弹窗/抽屉：180-220ms。
- 图表更新：300ms。
- Toast：3-5s 自动消失。

必须支持：

- 低动效模式：关闭非必要动画。
- 键盘焦点：输入框、按钮、列表项可见 focus。
- 弹窗层级：统一 z-index / Panel.ZIndex 规则。

## 10. 图标方向

`ui-ux-pro-max` 明确建议：不要使用 emoji 作为 UI 图标。

WPF 施工建议：

- 新增 `RelayBench.App/Resources/Icons.xaml`。
- 使用统一线性图标风格，接近 Lucide / Fluent 的 1.75-2px stroke。
- 常用图标包括：测试、对话、批量、应用、网络、历史、参数、图表、刷新、上传、下载、复制、删除、关闭、外链、警告。
- 图标尺寸固定：16px、18px、20px、24px 四档。
- 图标按钮必须有 tooltip。

需要替换的地方：

- 批量评测行内 `✦`。
- 模型拉取按钮里的 `⟳`。
- 应用卡片如果有 emoji 图标，全部换为矢量图标。
- 关于弹窗外链使用外链图标。

## 11. 状态体系方向

所有状态统一为 6 类：

| 状态 | 色彩 | 用途 |
|------|------|------|
| Idle | 灰色 | 未开始、空闲 |
| Running | 蓝色 | 正在测试、正在生成、正在扫描 |
| Success | 绿色 | 完成、通过、可用 |
| Warning | 橙色 | 兼容性未知、部分失败 |
| Error | 红色 | 失败、不可达、异常 |
| Disabled | 浅灰 | 不可操作 |

施工要求：

- 所有页面的 badge、Toast、日志、表格状态都使用同一套状态色。
- 错误必须给出下一步动作，如重试、查看 Trace、复制日志。
- 空状态必须给出入口，如开始扫描、开始测试、导入入口组。

## 12. 施工阶段建议

### 阶段 1：设计系统收敛

目标：

- 建立颜色、字号、间距、圆角、阴影、图标、状态 token。
- 清理过度渐变和过大圆角。
- 建立按钮、输入、下拉、表格、Tab、Expander、Toast、Overlay 样式。

文件：

- `RelayBench.App/Resources/WorkbenchTheme.xaml`
- `RelayBench.App/Resources/Icons.xaml`

验收：

- 新旧页面混用时不割裂。
- 所有控件默认样式已经接近大厂后台。

### 阶段 2：Shell 和命令区

目标：

- 重做主窗口 Shell。
- 统一页面头和命令区。
- 确定 Panel.ZIndex 层级表。

文件：

- `RelayBench.App/MainWindow.xaml`
- 各页面根布局。

验收：

- 页面切换稳定。
- 弹窗、抽屉、Toast 不互相遮挡。
- 窗口化不压缩主内容。

### 阶段 3：高频页面先行

顺序：

1. 单站测试。
2. 大模型对话。
3. 批量评测。

原因：

- 这 3 个页面最能体现产品价值。
- 这 3 个页面覆盖表单、图表、日志、聊天、表格、多模型等核心控件。

### 阶段 4：支撑页面跟进

顺序：

1. 应用接入。
2. 网络复核。
3. 历史报告。

原因：

- 应用接入需要资产卡和协议状态。
- 网络复核需要工具卡和诊断结果体系。
- 历史报告需要时间线和导出体验。

### 阶段 5：微交互和验收

目标：

- 补齐 loading、empty、error、toast、focus、keyboard、reduced motion。
- 做 1120x700、1280x720、1440x860、1920x1080 四档视觉验收。

## 13. 和上一份施工文档的关系
