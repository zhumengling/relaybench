请解释一下 WebSocket 和 SSE 的区别，各自的优缺点是什么？在什么场景下应该选择哪种方案？
多模型回答 21:08
GPT-4o 完成

WebSocket 是全双工通信协议，建立连接后客户端和服务端可以随时互相发送数据，适合实时聊天、多人协作编辑、在线游戏等需要双向通信的场景。

SSE (Server-Sent Events) 是单向的，只能服务端向客户端推送，基于 HTTP 协议，天然支持断线重连，适合通知推送、实时数据流、日志监控等场景。

// SSE 客户端示例
const es = new EventSource('/api/stream');
es.onmessage = (e) => console.log(e.data);

核心区别在于通信方向和协议层级：

特性WebSocketSSE
方向双向单向(服务端→客户端)
协议独立协议(ws://)基于HTTP
断线重连需自行实现内置支持
复杂度较高低

简单总结选型建议：

  • 需要双向实时通信（聊天室、协同编辑）→ WebSocket
  • 只需要服务端推送（通知、数据流）→ SSE
  • 需要兼容HTTP 代理/防火墙 → SSE 更友好
  • 需要二进制传输 → WebSocket

那如果我的应用同时需要这两种场景——既有实时双向的数据同步，又需要服务端单向推送通知，架构上应该怎么设计？
图片 architecture.png
GPT-4o 21:10

推荐采用混合架构，按通信需求分层：

1. 核心数据通道 — WebSocket

用于需要双向实时同步的场景，如文档协作、表单状态同步。客户端与服务端通过 WebSocket 保持长连接。

2. 通知推送通道 — SSE

用于服务端向客户端推送通知、系统消息等。SSE 基于 HTTP，天然穿透代理，断线自动重连。

// 架构示意
┌─────────────┐     WebSocket      ┌──────────┐
│   Client    │◄──────────────────►│  WS 服务  │  ← 实时双向
│             │     SSE            │          │
│             │◄───────────────────│ SSE 服务  │  ← 单向推送
└─────────────┘                    └──────────┘

3. 关键设计要点

  • WebSocket 连接用于状态同步，SSE 用于事件通知
  • 两种连接共享同一套认证机制（Token）
  • SSE 可作为 WebSocket 的降级方案
